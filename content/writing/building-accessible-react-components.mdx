# Building Accessible React Components That Scale

Accessibility is not a feature you bolt on at the end. It is a quality of the system itself. In large React codebases, the difference between an accessible product and an inaccessible one usually comes down to a handful of repeatable patterns.

## Start with semantic HTML

Before reaching for ARIA, use the right element. A button should be a `button`. A heading hierarchy should be logical. Native elements give you keyboard support, focus behaviour, and screen-reader announcements for free.

## Manage focus deliberately

When a modal opens, focus should move to the modal. When it closes, focus should return to the trigger. When a route changes, focus should move to the new heading. These small decisions make single-page apps usable for keyboard and screen-reader users.

## Test with real tools

Automated tests catch low-hanging fruit, but they do not replace manual testing with a keyboard and a screen reader. Build time into your workflow for both.

Accessibility is continuous. Embed it into code review, design review, and release criteria, and it becomes invisible to users in the best possible way.
